# PRD: Local Git Auto Commit & Push Daemon (Windows)

**Version:** 1.2
**Author:** Rogit
**Last Updated:** 2025-09-20

---

## 1. Objective

Design and implement a lightweight local automation service that monitors a specified
Git-tracked folder for file changes, batches them at periodic intervals, and automatically
commits and pushes updates to the remote repository under the configured Git author identity.

This system ensures that contribution activity is consistently captured on GitHub without
manual commit intervention — functioning as a silent background daemon on Windows.

---

## 2. Scope

### In Scope

* A single repository directory under Git version control.
* Continuous monitoring and periodic commit cycles.
* Auto-generation of meaningful commit messages.
* Authenticated remote pushes to GitHub.
* Failure resilience with minimal user intervention.
* Log rotation and retention policies.
* Network-awareness before push attempts.
* Graceful handling of Git state anomalies (merge, rebase, lock files).

### Out of Scope

* Multi-repo management.
* GUI or visualization dashboards.
* Merge conflict resolution (manual intervention required).
* File-level rollback or diff management.
* Cross-platform support (Windows-only for v1).

## 3. Core Requirements

### 3.1 Functional Requirements

| ID  | Requirement          | Description                                                                                                              | Priority |
| --- | -------------------- | ------------------------------------------------------------------------------------------------------------------------ | -------- |
| F1  | Folder Monitoring    | The daemon continuously monitors one specified folder for file modifications, additions, or deletions.                   | High     |
| F2  | Batch Commit         | Changes are batched and committed at scheduled intervals (default: every 5 minutes).                                     | High     |
| F3  | Auto Commit Message  | Commit messages are auto-generated in the format: `auto: updated files on YYYY-MM-DD HH:MM:SS`.                         | Medium   |
| F4  | Auto Push            | After a successful commit, changes are pushed to the remote branch.                                                      | High     |
| F5  | Author Attribution   | Commits must use globally configured Git author name and email that map to the user's GitHub identity.                   | High     |
| F6  | Logging              | Each run logs activity (timestamp, commit status, push status, errors) to a local `.logs/auto-commit.log`.               | Medium   |
| F7  | Conflict Handling    | If a remote push fails due to conflicts, the daemon suspends further actions and logs the error for manual intervention. | Medium   |
| F8  | Network-Aware Push   | Pushes should only attempt if a valid network connection exists. Skips push silently if offline.                         | Medium   |
| F9  | Repository Integrity | The daemon verifies `.git` folder exists and repo is not mid-merge/rebase before performing actions.                     | High     |
| F10 | Log Rotation         | Log files are automatically rotated when exceeding a configurable size threshold (default: 1MB).                         | Low      |
| F11 | Lock File Guard      | Prevents multiple instances of the daemon from running simultaneously via a PID lock file.                               | Medium   |
| F12 | Graceful Shutdown    | Handles Ctrl+C and system signals cleanly, removing lock files and logging shutdown.                                     | Medium   |
| F13 | File Size Filter     | Optionally skip staging files exceeding a configurable size limit to prevent large binary commits.                       | Low      |

| ID  | Requirement     | Description                                                                         |
| --- | --------------- | ----------------------------------------------------------------------------------- |
| N1  | Lightweight     | CPU and memory usage must remain negligible (<2% CPU, <50MB RAM).                   |
| N2  | Reliability     | No duplicate commits or infinite commit loops.                                      |
| N3  | Security        | Credentials are never stored in plaintext. Uses Git Credential Manager or SSH keys. |

## 4. System Design

### 4.1 Architecture

A single **background PowerShell process** orchestrates Git operations:

```
+---------------------------------------------------+
|                auto-commit.ps1                     |
|                                                    |
|  +----------+   +--------------+   +-----------+  |
|  |  Config   |-->|  Health Check |-->|  Stage &  |  |
|  |  Loader   |   |  & Safeguard  |   |  Commit   |  |
|  +----------+   +--------------+   +-----+-----+  |
|       |                                  |         |
|       v                                  v         |
|  +----------+                     +----------+     |
|  |  Logger   |<-------------------| Push      |     |
|  |  Module   |                    | Routine   |     |
|  +----------+                     +----------+     |
|                                                    |
|  +----------------------------------------------+  |
|  |         Timer Loop (Scheduler)                |  |
|  |    Executes cycle every N minutes             |  |
|  +----------------------------------------------+  |
+---------------------------------------------------+
```

1. **Timer Loop (Scheduler):**
   * Executes every *n* minutes (configurable).
   * Calls the commit routine.
   * Respects lock file to prevent concurrent execution.

2. **Config Loader:**
   * Reads `.env` file with key=value pairs.
   * Supports quoted and unquoted values.
   * Falls back to sane defaults if `.env` is missing.

3. **Health Check & Safeguard Module:**
   * Verifies `.git` directory exists.
   * Checks for merge/rebase in progress.
   * Detects Git index lock files.
   * Validates repo is not in detached HEAD state.

4. **Commit Routine:**
   * Detects changes using `git status --porcelain`.
   * If changes exist -> stage all -> commit with autogenerated message.

5. **Push Routine:**
   * Tests network connectivity before push attempt.
   * Attempts push to the configured remote branch.
   * Logs output and status.
   * If push fails due to upstream divergence, logs warning and halts.

6. **Logger Module:**
   * Timestamped log entries with severity levels.
   * Automatic log rotation at configurable threshold.
   * Console + file dual output.

### 4.2 Components

| Component                | Description                                                          |
| ------------------------ | -------------------------------------------------------------------- |
| `auto-commit.ps1`       | Main PowerShell daemon script containing all logic.                  |
| `.env`                   | Configuration file for repo path, interval, branch, and log path.    |
| `.env.example`           | Template configuration file committed to repo.                       |
| `.logs/auto-commit.log`  | Persistent log file (auto-rotated).                                  |
| `.logs/.lockfile`        | PID lock file preventing duplicate daemon instances.                 |
| `setup-scheduler.ps1`   | Helper script to register Windows Task Scheduler job.                |
| `.gitignore`             | Excludes .env, logs, and temp files from version control.            |

### 4.3 Configuration Parameters

| Parameter           | Default                              | Description                              |
| ------------------- | ------------------------------------ | ---------------------------------------- |
| `REPO_PATH`         | Current working directory            | Path to local Git repository.            |
| `BRANCH`            | `main`                               | Branch to push to.                       |
| `INTERVAL_MINUTES`  | `5`                                  | Time interval between commit cycles.     |
| `LOG_PATH`          | `.logs\auto-commit.log`              | Log file path.                           |
| `PUSH_ENABLED`      | `true`                               | Whether to push after commit.            |
| `MAX_LOG_SIZE_KB`   | `1024`                               | Max log file size before rotation (KB).  |
| `MAX_FILE_SIZE_MB`  | `50`                                 | Max individual file size to stage (MB).  |
| `NETWORK_CHECK_URL` | `https://github.com`                 | URL to ping for connectivity check.      |

## 5. Operational Flow

### 5.1 Startup Sequence

1. Load configuration from `.env` (with defaults fallback).
2. Validate repo path exists and contains `.git`.
3. Create log directory if missing.
4. Acquire lock file (exit if another instance running).
5. Register Ctrl+C handler for graceful shutdown.
6. Enter main timer loop.

### 5.2 Commit-Push Cycle

1. **Health Check:**
   * `.git` folder exists.
   * No merge/rebase/cherry-pick in progress.
   * No `index.lock` present.

2. **Change Detection:**
   * `git status --porcelain` for modified/untracked files.
   * If empty -> skip cycle.

3. **Stage & Commit:**
   * `git add -A`
   * Generate timestamped commit message.
   * `git commit -m "<message>"`

4. **Push (if enabled):**
   * Check network connectivity.
   * `git push origin <branch>`
   * On failure: log error, halt daemon.

5. **Log & Wait:**
   * Log cycle result (success/skip/error).
   * Sleep for configured interval.
   * Repeat.

### 5.3 Shutdown Sequence

1. Ctrl+C or system signal received.
2. Log shutdown event.
3. Remove lock file.
4. Exit cleanly.

---

## 6. Safeguards

| Check                | Mechanism                                                  |
| -------------------- | ---------------------------------------------------------- |
| Merge in progress    | Skip cycle if `.git/MERGE_HEAD` exists.                    |
| Unresolved rebase    | Skip if `.git/rebase-apply` or `.git/rebase-merge` exists. |
| Cherry-pick active   | Skip if `.git/CHERRY_PICK_HEAD` exists.                    |
| Git index locked     | Skip if `.git/index.lock` exists.                          |
| Network unavailable  | Skip push, retry next cycle.                               |
| Credential missing   | Fail gracefully, log warning.                              |
| Duplicate instance   | Lock file prevents concurrent runs.                        |
| Large file detected  | Skip files exceeding MAX_FILE_SIZE_MB.                     |

## 9. Risk Assessment

| Risk                                        | Impact | Mitigation                                           |
| ------------------------------------------- | ------ | ---------------------------------------------------- |
| Infinite commit loop due to generated files | High   | `.gitignore` enforcement and commit diff validation  |
| Accidental overwrite due to force push      | Medium | Disallow `--force` in automation                     |
| GitHub credential expiry                    | Medium | Use credential manager or SSH-based authentication   |
| Network unavailability                      | Low    | Retry silently on next interval                      |
| Large binary file commits                   | Low    | File size filter (MAX_FILE_SIZE_MB)                  |
| Multiple daemon instances                   | Medium | PID lock file prevents concurrent execution          |
| Log file disk exhaustion                    | Low    | Automatic log rotation at size threshold             |
