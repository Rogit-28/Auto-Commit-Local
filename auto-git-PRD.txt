# PRD: Local Git Auto Commit & Push Daemon (Windows)

**Version:** 1.0
**Author:** Rogit
**Last Updated:** 2025-09-16

---

## 1. Objective

Design and implement a lightweight local automation service that monitors a specified
Git-tracked folder for file changes, batches them at periodic intervals, and automatically
commits and pushes updates to the remote repository under the configured Git author identity.

This system ensures that contribution activity is consistently captured on GitHub without
manual commit intervention — functioning as a silent background daemon on Windows.

---

## 2. Scope

### In Scope

* A single repository directory under Git version control.
* Continuous monitoring and periodic commit cycles.
* Auto-generation of meaningful commit messages.
* Authenticated remote pushes to GitHub.
* Failure resilience with minimal user intervention.

### Out of Scope

* Multi-repo management.
* GUI or visualization dashboards.
* Merge conflict resolution (manual intervention required).
* File-level rollback or diff management.
* Cross-platform support (Windows-only for v1).

## 3. Core Requirements

### 3.1 Functional Requirements

| ID  | Requirement          | Description                                                                                                    | Priority |
| --- | -------------------- | -------------------------------------------------------------------------------------------------------------- | -------- |
| F1  | Folder Monitoring    | The daemon continuously monitors one specified folder for file modifications, additions, or deletions.         | High     |
| F2  | Batch Commit         | Changes are batched and committed at scheduled intervals (default: every 5 minutes).                           | High     |
| F3  | Auto Commit Message  | Commit messages are auto-generated in the format: `auto: updated files on YYYY-MM-DD HH:MM:SS`.               | Medium   |
| F4  | Auto Push            | After a successful commit, changes are pushed to the remote branch.                                            | High     |
| F5  | Author Attribution   | Commits must use globally configured Git author name and email that map to the user's GitHub identity.         | High     |
| F6  | Logging              | Each run logs activity (timestamp, commit status, push status, errors) to a local `.logs/auto-commit.log`.     | Medium   |
| F7  | Conflict Handling    | If a remote push fails due to conflicts, the daemon suspends further actions until manual intervention.        | Medium   |
| F8  | Network-Aware Push   | Pushes should only attempt if a valid network connection exists.                                               | Medium   |
| F9  | Repository Integrity | The daemon verifies `.git` folder exists and repo is not mid-merge or rebasing before performing actions.      | High     |

### 3.2 Non-Functional Requirements

| ID  | Requirement     | Description                                                                         |
| --- | --------------- | ----------------------------------------------------------------------------------- |
| N1  | Lightweight     | CPU and memory usage must remain negligible (<2% CPU, <50MB RAM).                   |
| N2  | Reliability     | No duplicate commits or infinite commit loops.                                      |
| N3  | Security        | Credentials are never stored in plaintext. Uses Git Credential Manager or SSH keys. |
| N4  | Compatibility   | Works on Windows 10/11 PowerShell 5.1+ environment.                                |
| N5  | Maintainability | Configurable via `.env` file defining interval, repo path, and branch.              |

## 4. System Design

### 4.1 Architecture

A single **background PowerShell process** orchestrates Git operations:

1. **Timer Loop (Scheduler):**
   * Executes every *n* minutes (configurable).
   * Calls the commit routine.

2. **Commit Routine:**
   * Verifies repo health.
   * Detects changes using `git status --porcelain`.
   * If changes exist -> stage all -> commit with autogenerated message.

3. **Push Routine:**
   * Attempts push to the current branch.
   * Logs output and status.
   * If push fails due to upstream divergence, logs warning and halts further runs.

### 4.2 Components

| Component              | Description                                                       |
| ---------------------- | ----------------------------------------------------------------- |
| `auto-commit.ps1`     | Main PowerShell daemon script that runs in loop.                  |
| `.env`                 | Configuration file for repo path, interval, branch, and log path. |
| `auto-commit.log`      | Persistent log file stored under `.logs/`.                        |
| Windows Task Scheduler | Service to trigger the PowerShell script on login or at interval. |

### 4.3 Configuration Parameters

| Parameter          | Default                              | Description                          |
| ------------------ | ------------------------------------ | ------------------------------------ |
| `REPO_PATH`       | Current working directory            | Path to local Git repository.        |
| `BRANCH`          | `main`                               | Branch to push to.                   |
| `INTERVAL_MINUTES`| `5`                                  | Time interval between commit cycles. |
| `LOG_PATH`        | `.logs\auto-commit.log`              | Log file path.                       |
| `PUSH_ENABLED`    | `true`                               | Whether to push after commit.        |

---

## 5. Operational Flow

### 5.1 Commit-Push Cycle

1. Daemon starts (manual launch or scheduled task).
2. Checks repo validity:
   * `.git` folder exists.
   * `git status` returns clean working tree state.
3. Detects file modifications (`git status --porcelain`).
4. If non-empty:
   * `git add -A`
   * `git commit -m "auto: updated files on <timestamp>"`
   * `git push origin <branch>`
5. Logs result.
6. Waits for `INTERVAL_MINUTES` and repeats.

## 6. Stack & Dependencies

* **Language:** PowerShell 5.1+ (ships with Windows 10/11)
* **Runtime:** Windows 10/11
* **Version Control:** Git for Windows (2.30+)
* **Auth:** SSH key or Git Credential Manager Core
* **Scheduler:** Windows Task Scheduler (optional, for auto-start)
* **External Dependencies:** None (pure PowerShell + Git CLI)
